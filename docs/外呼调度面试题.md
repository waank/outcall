## 外呼系统项目面试常见问题整理
### 一、系统架构与设计类
#### 1. 请介绍一下这个外呼系统的整体架构？
考察点：系统设计能力、分层架构理解

回答思路（STAR法则）：

+ Situation：项目是一个面向企业通信自动化的智能外呼调度系统，需要支持高并发、可伸缩的外呼调度
+ Task：设计一个能支撑大规模并发外呼的系统架构
+ Action：采用"控制器-服务-数据访问-存储/缓存"的分层架构，结合多线程池与限流策略
+ Result：实现了高吞吐、低延迟的外呼调度流水线，通过任务-分组-队列三层模型管理外呼流程

---

#### 2. 为什么采用任务-分组-队列三层模型？这样设计有什么好处？
考察点：业务建模能力、设计权衡

回答思路：

+ 分层解耦：任务层管理整体外呼任务，分组层实现批量调度，队列层处理具体号码
+ 灵活调度：支持择时外呼和预测式外呼两种策略
+ 状态隔离：每层有独立状态管理，便于故障恢复和重试
+ 性能优化：分组批量处理减少数据库压力，队列级并发提升吞吐

---

### 二、并发与多线程类
#### 3. 系统中使用了哪些线程池？为什么要这样设计？
考察点：线程池隔离、并发设计

回答思路：

+ queueGroupthreadPool：处理队列组级别任务
+ retryThreadPool：处理异常重试任务（使用CallerRunsPolicy保护关键路径）
+ outCallThreadPool：主外呼执行线程池
+ largeMakeCallThreadPool：大租户专用线程池，避免资源争抢
+ planTaskThreadPool：计划类任务调度
+ commonMakeCallThreadPool：通用外呼任务

设计原因：

+ 线程池隔离避免不同阶段相互影响
+ 针对不同场景选择不同拒绝策略
+ 支持动态调整核心/最大线程数

---

#### 4. 线程池的拒绝策略是如何选择的？
考察点：拒绝策略理解、实际应用

回答思路：

+ DiscardPolicy（丢弃策略）：用于非关键路径（如队列组处理、外呼执行），避免系统过载
+ CallerRunsPolicy（调用方执行）：用于重试线程池，保护关键路径，降低峰值冲击
+ 选择原则：根据业务重要性权衡，非关键路径丢弃保稳定，关键路径调用方执行保可用

---

### 三、分布式系统类
#### 5. 分布式锁是如何实现的？有哪些关键设计？
考察点：Redis分布式锁、并发控制

回答思路：

+ 加锁：使用Redis `setIfAbsent`（setNx）原子操作，设置过期时间
+ 解锁：使用Lua脚本，仅当键值匹配时删除，避免误删他人锁
+ 续期机制：定时扫描本地维护的锁信息表，当剩余时间低于阈值时调用expire续期
+ 死锁预防：所有锁均设置过期时间，结合续期机制避免长时间占用

应用场景：

+ 任务规划锁：TaskPlanService对单任务加锁，避免多实例同时规划
+ 队列组规划锁：区分普通组与择时组
+ 组执行锁：对具体队列组加锁，结合存在性检查实现异常恢复

---

#### 6. 如何保证分布式锁的安全性？
考察点：分布式锁细节、边界情况处理

回答思路：

+ 值匹配释放：解锁时使用Lua脚本判断锁值一致再删除
+ 自动续期：长时间持有锁时定时续期，避免过期导致竞态
+ 异常容错：续期失败且Redis中不存在该key时，清理本地续期记录
+ 锁粒度设计：任务级锁（短锁约60s）、组级锁（针对队列组执行阶段）

---

### 四、缓存与数据库类
#### 7. Redis缓存是如何使用的？
考察点：缓存设计、原子操作

回答思路：

+ 队列组缓存：使用Redis列表存储队列组编码
+ 原子操作：使用Lua脚本执行LPUSH批量入队与EXPIRE设置，保证原子性
+ 键空间设计：区分私有组与公共组两类键空间，便于隔离与清理
+ 缓存上限控制：通过调度参数限制缓存组数量上限，避免缓存膨胀

---

#### 8. 数据库表是如何设计的？索引策略是什么？
考察点：数据库设计、索引优化

回答思路：

+ 核心表：
    - `cc_outcall_queue`：呼叫名单表，存储被叫号码和呼叫状态
    - `cc_outcall_queue_group`：队列分组表
    - `cc_outbound_call_task`：外呼任务表
    - `cc_outbound_timing_info`：择时信息表
    - `cc_outbound_call_task_rules`：任务规则表
+ 索引策略：
    - 唯一索引保证数据唯一性
    - 复合索引优化高频查询场景
    - 状态字段索引支持快速筛选

---

### 五、限流与调度类
#### 9. 限流机制是如何实现的？
考察点：限流算法、流量控制

回答思路：

+ 等待机制：基于配置的等待时长与轮询间隔，持续检查限流状态
+ 超时处理：超过等待时长仍未解除限流则判定超时
+ 与分布式锁协作：进入限流等待前先尝试获取分布式锁
+ 设计建议：可实现全局限流、组内限流、CPS限流等多维度策略

---

#### 10. 任务调度算法是如何设计的？
考察点：调度算法、时间窗口控制

回答思路：

+ 任务分页：分页扫描进行中的任务，逐条校验任务状态与时间窗口
+ 队列组轮询：从Redis列表右侧批量弹出队列组代码
+ 时间窗口控制：任务与队列组均需满足"当前可呼时间段"
+ 负载均衡：多线程池隔离、队列长度阈值、限流等待与重试策略

---

### 六、异常处理与容错类
#### 11. 系统如何处理异常和重试？
考察点：异常处理、重试策略

回答思路：

+ 异常分类：业务异常（状态非法、时间窗口不匹配）、系统异常（线程池满、Redis锁冲突）、网络异常
+ 重试策略：
    - 最大重试次数控制，超过则标记STOP
    - 异常时生成重试组，提交到retryThreadPool
    - 限流等待支持超时控制
+ 降级策略：队列积压/线程池满时将队列状态置为WAITING

---

#### 12. 如何保证系统的高可用？
考察点：系统稳定性、容错设计

回答思路：

+ 线程池隔离：不同职责使用不同线程池，避免相互影响
+ 限流与背压：基于限流服务与线程池队列长度进行保护
+ 分批与并行：任务计划阶段采用分批与并行子批次
+ 缓存与锁：Redis缓存与分布式锁用于运行时批取与组级互斥
+ 监控与日志：定时输出线程池状态，便于及时发现瓶颈
